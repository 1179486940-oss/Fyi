# Multi-Agent System 说明文档

## 1. 项目概览

本项目位于 `multi_agent_system/`，是一套基于“主路由 Agent + 多个子 Agent”的多智能体系统骨架，当前版本以 **mock/stub** 方式跑通整体链路，便于后续替换为真实接口。

当前系统的核心目标是：
- 接收用户自然语言请求
- 由主路由 Agent 识别意图并拆分任务
- 调用对应的子 Agent 执行业务
- 支持多轮会话、长期记忆触发、上下文 checkpoint、写操作确认、多模态输入、产物输出
- 最终返回结构化结果给前端或调用方

## 3. 系统整体执行流程

### 3.1 总体流程

用户请求进入系统后，整体会经过以下步骤：

1. **入口接收请求**
   - 由 `main.py` 或未来的 API 层接收 `session_id`、`user_id`、`query`、`attachments`。

2. **RouterAgent 总控处理**
   - 文件：`agents/router_agent.py`
   - 负责多模态注入、记忆触发、checkpoint 创建、澄清判断、任务规划、调用子 Agent、聚合结果。

3. **Guardrails 进行安全与边界判断**
   - 文件：`core/guardrails.py`
   - 判断问题是否过短、是否需要澄清、是否可能是写操作。

4. **SessionManager 创建 checkpoint**
   - 文件：`core/session_manager.py`
   - 为当前轮对话建立 checkpoint，支持后续回溯恢复。

5. **MemoryManager 检查长期记忆触发词**
   - 文件：`core/memory_manager.py`
   - 如果用户输入中包含“记住 / 别忘了 / 存起来”等关键词，就写入长期记忆。
   - 如果包含“忘了吧 / 不用记住”，则删除最近一条长期记忆。

6. **RouterAgent 规划任务**
   - 根据 query 中的关键词，规划调用哪些子 Agent。
   - 当前支持：
     - 数据查询/改数
     - 图表生成
     - 报表生成
     - PPT 生成
     - 兜底知识问答

7. **执行一个或多个子 Agent**
   - 每个子 Agent 接收统一的 `AgentContext`。
   - 执行完成后返回统一的 `AgentResult`。

8. **聚合结果**
   - Router 将多个 Agent 的 summary 拼接成最终 summary。
   - 返回结构化字典结果。

## 4.2 `core/models.py` —— 统一数据模型

文件：`core/models.py`

定义系统内所有核心结构体，保证不同模块之间传输的数据格式统一。

表示一条知识库检索结果。
字段：
- `kb_type`
- `content`
- `score`
- `metadata`

#### 4）`ConfirmationPayload`
表示写操作确认卡片的数据。
字段：
- `confirmation_id`
- `operation`
- `summary`
- `sql`
- `preview_rows`
- `metadata`

#### 5）`AgentEvent`
表示一条事件。
类型包括：
- `answer`
- `clarification`
- `confirmation`
- `artifact_ready`
- `error`
- `trace`

#### 6）`Artifact`
表示一个生成的产物文件。
字段：
- `name`
- `artifact_type`
- `path`
- `url`
- `metadata`

#### 7）`AgentResult`
每个子 Agent 的统一返回结构。
字段：
- `status`
- `agent_name`
- `summary`
- `data`
- `table_preview`
- `artifacts`
- `confirmation_required`
- `clarification_required`
- `events`
- `trace`
- `confirmation_payload`

#### 8）`RouterTask`
表示 Router 规划出来的一个任务。
字段：
- `agent_name`
- `user_query`
- `dependency_on`
- `metadata`

#### 9）`RouterState`
表示主路由执行过程中的完整上下文。
字段：
- `session_id`
- `user_id`

- `attachments`
- `normalized_query`
- `multimodal_context`
- `memory_context`
- `retrieval_context`
- `planned_tasks`
- `events`
- `checkpoints`

### 使用方法
系统各模块间统一传这些 dataclass；新增功能时优先复用这些模型，而不是传裸 dict。

## 4.3 `core/session_manager.py` —— 会话管理与 checkpoint

文件：`core/session_manager.py`

### 作用
管理会话历史与 checkpoint 恢复。

### 当前实现功能

#### 1）`append_turn(session_id, role, content)`
向会话中追加一轮消息，并自动截断到最近 `memory_limit` 轮。

#### 2）`get_recent_turns(session_id)`
读取该 session 最近保留的消息。

#### 3）`create_checkpoint(state)`
根据当前 `RouterState` 生成一个 checkpoint id，并保存快照。

#### 4）`restore_checkpoint(session_id, checkpoint_id)`
恢复指定 checkpoint 对应的状态。

### 实现流程
1. Router 收到 query 后创建 `RouterState`
2. 调用 `create_checkpoint(state)`
3. 获得 checkpoint id
4. 后续如果用户需要“回到刚才的问题”，就可通过 `restore_checkpoint()` 恢复

### 使用方法

```python
checkpoint_id = session_manager.create_checkpoint(state)
old_state = session_manager.restore_checkpoint(session_id, checkpoint_id)
```

## 4.4 `core/memory_manager.py` —— 长期记忆管理

文件：`core/memory_manager.py`

### 作用
负责长期记忆触发词识别、写入、删除、查询。

### 当前触发词
写入触发词：
- `记住`
- `长久记住`
- `别忘了`
- `以后要用`
- `存起来`
- `这个很重要`
- `以后会用到`

删除触发词：
- `不用记住`
- `忘了吧`

### 当前实现功能

#### 1）`detect_memory_action(query)`
识别当前 query 是写记忆、删记忆还是不操作。

#### 2）`write_long_term_memory(session_id, content, keyword)`
写入一条长期记忆，返回 `MemoryRecord`。

#### 3）`delete_last_long_term_memory(session_id)`
删除该 session 最近一条长期记忆。

#### 4）`get_long_term_memory(session_id)`
取出该 session 下全部长期记忆。

### 实现流程
1. Router 拿到用户 query
2. 调用 `detect_memory_action()`
3. 如果是写入，则保存一条 `MemoryRecord`
4. 如果是删除，则删除最近一条
5. 将结果作为 trace event 写入 `RouterState.events`

### 使用示例

用户输入：
```text
记住，销售周报优先用东区模板
```

系统行为：
- 识别到关键词“记住”
- 将整句写入长期记忆
- 记录一条 trace event

## 4.6 `core/auth_manager.py` —— 鉴权管理

文件：`core/auth_manager.py`

### 作用
模拟子系统权限与字段级屏蔽规则。

### 当前实现逻辑
内置两类用户：
- `demo-user`：全部权限开启
- `restricted-user`：只能访问部分能力

### 核心方法
`authorize(user_id, subsystem)`

返回 `AuthDecision`：
- `allowed`
- `reason`
- `hidden_fields`

### 实现流程
1. 子 Agent 运行前先调用 `authorize()`
2. 如果 `allowed=False`，直接返回 error
3. 如果允许访问，但存在字段限制，就在结果返回前过滤掉对应字段

### 使用示例

```python
auth = auth_manager.authorize("demo-user", "data")
if not auth.allowed:
    ...
```

## 4.8 `core/confirmation_service.py` —— 写操作确认服务

文件：`core/confirmation_service.py`

### 作用
保存待确认的写操作状态，支持确认、拒绝、查询。

### 当前功能

#### 1）`request(payload)`
登记一条待确认操作。

#### 2）`resolve(confirmation_id, approved)`
将某条待确认操作标记为通过或拒绝。

#### 3）`get(confirmation_id)`
获取某条待确认操作的当前状态。

### 当前实现流程
1. Query Agent 识别到写操作
2. 构造 `ConfirmationPayload`
3. `ConfirmationService.request(payload)` 保存状态
4. 返回 `confirmation_required` 给前端
5. 前端确认后，未来可调用 `resolve()` 继续执行

> 当前版本已搭好确认挂起机制的数据结构，但“确认后继续执行 SQL”这一段还是下一步要补全的逻辑。

## 4.9 `core/query_service.py` —— 数据查询与 SQL 规划

文件：`core/query_service.py`

### 作用
根据用户 query 生成 mock 查询计划 `QueryPlan`。

### 返回结构
`QueryPlan` 包含：
- `sql`
- `operation`
- `preview_rows`
- `result_rows`

### 当前实现逻辑

#### 1）写操作分支
如果命中写操作关键词：
- 构造一条 mock UPDATE SQL
- 返回受影响数据预览
- operation = `UPDATE`

#### 2）销售数据查询分支
如果 query 中包含“sales”或“销售”：
- 构造销售数据查询 SQL
- 返回销售数据样例

#### 3）默认状态表查询分支
否则：
- 返回 `status_table` 的默认查询 SQL 和 preview rows

### 使用方法

```python
plan = query_service.build_plan(query)
print(plan.sql)
print(plan.operation)
```

## 5. 主路由 Agent 说明

## 5.1 `agents/router_agent.py`

文件：`agents/router_agent.py`

### 作用
系统总控入口，负责整个任务流转。

### 初始化时做的事
RouterAgent 在 `__init__()` 中初始化：
- `LLMProvider`
- `SessionManager`
- `MemoryManager`
- `KnowledgeManager`
- `AuthManager`
- `Guardrails`
- `ConfirmationService`
- `QueryService`
- 5 个子 Agent 实例

### 核心方法：`handle(session_id, user_id, query, attachments=None)`

这是主入口方法。

### 详细执行流程

#### 第 1 步：处理附件与多模态内容
```python
multimodal_context = self.llm_provider.multimodal_extract(attachments)
normalized_query = self._normalize_query(query, multimodal_context)
```
- 如果传入了附件路径，先调用多模态抽取
- 再把抽取文本拼接到 query 后面

#### 第 2 步：初始化 RouterState
保存本轮请求的全部上下文信息。

#### 第 3 步：处理长期记忆触发词
```python
memory_decision = self.memory_manager.detect_memory_action(query)
```
- 写入长期记忆
- 删除长期记忆
- 或什么都不做

#### 第 4 步：创建 checkpoint
```python
checkpoint_id = self.session_manager.create_checkpoint(state)
```
保存当前上下文快照。

#### 第 5 步：做澄清判断
```python
guardrail = self.guardrails.evaluate_query(query)
```
如果问题过短，则直接返回：
- `status = clarification_required`
- 一条 clarification event

#### 第 6 步：任务规划
```python
state.planned_tasks = self._plan_tasks(query)
```
当前依据关键词规划任务：
- 包含“查 / 销售 / 状态 / update / delete / insert” -> `data_query_update`
- 包含“图 / chart / plot / 折线 / 柱状” -> `data_graph`
- 包含“报表 / excel / report” -> `data_report`
- 包含 `ppt` -> `ppt_generate`
- 都不匹配 -> `fallback`

#### 第 7 步：顺序执行子任务
依次构造 `AgentContext`，调用对应子 Agent 的 `run()`。

#### 第 8 步：聚合结果
将所有子 Agent 的 `summary` 用 `|` 拼接，形成最终 summary。

### 使用方法

```python
from agents.router_agent import RouterAgent

router = RouterAgent()
result = router.handle(
    session_id="demo-session",
    user_id="demo-user",
    query="查询上个月销售数据并生成图表和PPT",
    attachments=[]
)
print(result)
```

## 6.1 `DataQueryUpdateAgent` —— 数据查询 / 改数 Agent

文件：`agents/sub_agents/data_query_update_agent.py`

### 作用
处理数据表查询与写操作确认，是整个系统最关键的业务 Agent。

### 当前实现流程

#### 第 1 步：鉴权
```python
auth = self.deps.auth_manager.authorize(context.user_id, self.subsystem)
```
如果无权限，直接返回 error。

#### 第 2 步：生成 QueryPlan
```python
plan = self.query_service.build_plan(context.query)
```
得到：
- SQL
- 操作类型
- 预览数据

#### 第 3 步：字段过滤
根据 `auth.hidden_fields` 过滤掉不允许返回的字段。

#### 第 4 步：判断是否是写操作
如果 `plan.operation != "SELECT"`：
- 生成 `ConfirmationPayload`
- 调用 `confirmation_service.request(payload)`
- 返回 `confirmation_required`

#### 第 5 步：如果是查询操作
直接返回：
- `status = success`
- SQL
- `table_preview`

### 当前返回结果类型

#### 查询场景
返回：
- `status = success`
- 查询 SQL
- 表格预览数据

#### 改数场景
返回：
- `status = confirmation_required`
- 变更预览数据
- `confirmation_payload`

### 使用示例

#### 查询数据
```python
result = router.handle(
    session_id="s1",
    user_id="demo-user",
    query="查询状态表"
)
```

#### 修改数据
```python
result = router.handle(
    session_id="s2",
    user_id="demo-user",
    query="把状态表里面运行中的状态改成终止"
)
```

这时系统不会立刻执行，而是返回确认信息。

## 13. 当前版本的限制

当前是第一版骨架，已打通框架，但还存在这些限制：

1. **全部是 mock/stub 实现**
   - 还未接真实数据库
   - 还未接真实 RAGFlow
   - 还未接真实对象存储
   - 还未接真实 PPT 服务
   - 还未接真实多模态模型

2. **写操作确认只做到“挂起返回确认信息”**
   - 还没实现“用户确认后继续执行”的完整闭环 API

3. **多意图链路目前是顺序执行**
   - 还没有更复杂的 DAG 调度

4. **Graph / Report / PPT 目前还没有真正消费 Query Agent 的输出数据**
   - 现在是先把流程骨架和 artifact 机制跑通
   - 下一步要加强数据复用链路

5. **路由目前主要还是关键词规划**
   - `LLMProvider.structured_intent()` 已留好入口
   - 后续可以切成真正的规则 + 结构化 LLM 分类

6. **当前环境无法本地运行测试**
   - 因为机器里没有 `python` 或 `py`

## 14. 后续建议迭代方向

建议后续按以下顺序增强：

1. **收敛目录规范**
   - 解决 `data_query_update_agent.py` 与 `data_query_update_agent/` 同名目录的维护问题

2. **强化 Query -> Graph/Report/PPT 的数据传递**
   - 让后续 Agent 真正复用前一个 Agent 的输出

3. **补齐确认闭环**
   - 前端确认 -> 后端 resolve -> 继续执行 SQL

4. **接真实接口**
   - RAGFlow
   - 鉴权接口
   - 数据查询接口 / DB
   - PPT 服务
   - 多模态模型

5. **加强 Router 的结构化意图识别**
   - 让规划不只依赖关键词

6. **加入 API 层**
   - 对外暴露 HTTP / WebSocket / SSE
